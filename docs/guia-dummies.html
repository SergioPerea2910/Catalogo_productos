Guia Dummies del Proyecto

Esta guia esta pensada para repasar rapido antes de la entrevista. La idea no es hablar como libro, sino entender que hace cada pieza y poder explicarla con palabras normales.

1. Que es este proyecto

Es una aplicacion fullstack para administrar un catalogo de productos digitales.

  • Backend: Laravel en PHP
  • Frontend: Next.js
  • Base de datos: SQLite
  • Documentacion: OpenAPI

Frase corta para entrevista: "Construí un CRUD de productos digitales con backend en Laravel y frontend en Next, con validación, protección básica de endpoints, pruebas y documentación."

2. Como fluye una peticion

Frontend -> Next API route -> Laravel API -> Base de datos -> respuesta

El navegador no llama directo al backend protegido. Primero le pega a Next y Next hace de intermediario para no exponer la API key en el cliente.

3. Backend explicado facil

routes/api.php: define que URLs existen.

Controller: recibe la petición y decide a quién llamar.

Request: valida que los datos lleguen bien.

Service: tiene la lógica de negocio.

Repository: habla con la base de datos.

Model: representa la tabla products.

Migration: dice cómo se crea la tabla.

Middleware: revisa si la API key es correcta.

Mapa mental corto: ruta -> controller -> request -> service -> repository -> model -> base de datos

4. Campos del producto

  • sku: codigo interno unico
  • name: nombre del producto
  • description: descripcion
  • price: precio
  • type: tipo de producto
  • is_active: activo o inhabilitado

id, created_at y updated_at los maneja Laravel.

5. Por que el DELETE no borra

Porque en esta solucion el borrado es logico, no fisico. En vez de eliminar el registro, se cambia is_active a false.

Frase para entrevista: "Preferí inhabilitación lógica para no perder trazabilidad y evitar borrar datos que podrían servir después."

6. Por que el listado solo muestra activos

Si DELETE solo inhabilita, entonces lo coherente es que el catálogo principal muestre únicamente productos activos. Los inactivos siguen en base de datos, pero no aparecen en la vista principal ni en GET /api/products.

7. Seguridad basica

Los endpoints de productos piden un header X-API-KEY. Si la llave no llega o es incorrecta, Laravel responde 401 Unauthorized.

8. Validacion

La validación vive en clases separadas:

  • StoreProductRequest
  • UpdateProductRequest

Eso permite que el controller quede limpio y no tenga todas las reglas mezcladas.

9. Frontend explicado facil

app/page.tsx: pantalla principal.

useState: guarda datos en memoria.

useEffect: carga productos al abrir la página.

handleSubmit(): crea o actualiza un producto.

loadProducts(): vuelve a consultar el listado.

app/api/products: proxy interno de Next.

lib/backend.ts: helper que llama a Laravel.

.env.local: guarda la URL del backend y la API key.

10. Por que la tabla se actualiza sola

Cuando creas, editas o inhabilitas, el frontend vuelve a ejecutar loadProducts(). Eso trae la lista actualizada del backend y React vuelve a pintar la pantalla. 

guardar -> respuesta OK -> loadProducts() -> setProducts() -> React repinta

11. Pruebas

El backend tiene pruebas para verificar:

  • creación de producto
  • errores de validación
  • protección por API key
  • listado solo de productos activos

12. Docker en palabras normales

Docker sirve para que el proyecto corra igual en otra máquina sin depender de tener PHP instalado localmente. 

docker compose up --build

Ese comando construye el backend, instala dependencias, corre migraciones y levanta Laravel.

13. OpenAPI / Swagger

El archivo docs/openapi.yaml describe la API: rutas, request body, respuestas y autenticación. No ejecuta lógica; solo documenta.

14. Preguntas tipicas y respuestas cortas

  • Por que separaste por capas?
    Para separar responsabilidades y hacer el código más fácil de mantener y probar.
  • Por que SQLite?
    Porque para una prueba técnica es rápido de montar y suficiente para demostrar persistencia.
  • Que mejorarías en producción?
    Base de datos más robusta, caché, paginación, monitoreo y autenticación más fuerte.

15. Comandos utiles

docker compose up --build
cd frontend
npm install
npm run dev

cd backend
docker run --rm -v "${PWD}:/app" -w /app composer php artisan test

16. Resumen final para que no te enredes

Backend: recibe, valida, aplica reglas y guarda. Frontend: muestra la información, envía formularios y refresca la vista. Next hace de puente para no exponer la API key. Docker facilita ejecutar el backend en cualquier máquina. OpenAPI documenta la API. Los tests demuestran que lo importante sí funciona.
